认知更新

  • ACK 路由架构：printf() 只走串口，WiFi 客户端永远收不到响应。解决方案是回调架构 — wallpaper 组件通过函数指针发送 ACK，main.c 设置回调为 wifi_ctrl_send()，避免循环依赖
  • 缓冲区溢出是静默的：chunk_buf 256 字节，Base64 解码超限不会报错，只是截断。工程级做法是增大到 1024 并放在模块级（非栈），便于清理
  • PSRAM 泄漏的隐蔽性：函数级 static 变量（重组缓冲区）在传输中断时永远无法释放。必须提升为模块级变量，并在新传输开始时主动清理
  • 超时保护是必须的：s_uploading 标志无超时 → 屏幕冻结 + SPI 总线死锁。60 秒自动复位是工程底线

行为变更

  • 从"能跑就行"到"工程级修复"：每个 bug 都要找到根因，不是绕过
  • 通信协议设计时先定义 ACK 路由方案，再写代码
  • 所有 volatile 标志必须有超时保护
  • PSRAM 分配必须有配对的释放路径

建立的新习惯

  • Firmware 组件间通信用回调而非直接调用，避免循环依赖
  • Base64 解码缓冲区统一为模块级 static，便于跨函数清理
  • 每个上传/传输流程都要有超时自动复位机制
  • 修复后先在串口测试 5 个 chunk 验证 ACK 路由，再做完整测试

关键教训

  • 通信失败的根因往往不是"发不出去"，而是"收不到回" — 先检查 ACK 路由
  • printf() 在嵌入式环境下不等于"发送到客户端" — 必须显式选择通道
  • 工程级修复 = 根因分析 + 架构改进 + 超时保护 + 资源清理
