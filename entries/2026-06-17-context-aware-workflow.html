Context-Aware Workflow Controller

ESP32 BLE GATT Python WinRT 工作流 产品设计
◫ AI-Agent-Deck

认知更新

  • 产品定位升级: 从"宏键盘"进化为"Context-Aware Workflow Controller"，核心价值是减少认知负担而非固定快捷键
  • BLE 双服务架构: ESP32 同时运行 HID 服务和自定义 GATT 服务时，必须使用统一回调分发，不能各自注册
  • BLE 长写入机制: WinRT API 对大数据使用 Prepare Write + Execute Write，ESP32 需要在 is_prep=1 时累积数据
  • Windows BLE 缓存: 已配对设备的 GATT 服务列表会被 Windows 缓存，修改固件后需要删除设备重新配对
  • 串口 vs BLE: 开发阶段串口通信更稳定可靠，BLE 用于最终产品形态

行为变更

  • 先规划后编码: 本次先完成完整产品规划（4周开发计划），再按优先级编码
  • 双通道通信: 同时维护串口（调试）和 BLE（产品）两条通信通道
  • 模块化 PC Manager: 将窗口检测、Profile管理、设备通信、工作流协调拆分为独立模块
  • 行为记录预留: 架构中预留 SQLite 行为日志和 AI 推荐接口

关键教训

  • BLE GATT 回调冲突: 两个 GATT 服务不能各自注册回调，后注册的会覆盖前一个。必须用统一回调 + gatts_if 过滤
  • UUID 格式对齐: ESP32 用 16-bit UUID 会自动转为 Bluetooth Base UUID，PC 端必须用匹配的 128-bit 格式
  • Handle 偏移: ESP32 返回的 char handle 是声明句柄，实际值句柄 = 声明句柄 + 1
  • Windows 自动配对: BLE 设备配对后 Windows 会自动连接，导致 Python 扫描不到设备
  • 产品思维: 不是"能用就行"，而是"用户无需思考"——设备主动适应用户，而非用户适应设备

技术实现

  • 窗口检测: win32gui + psutil，500ms 轮询，检测进程名和窗口标题
  • Profile 系统: 6个内置 Profile (VSCode/Blender/KiCad/Word/Chrome/Default)，JSON 存储
  • 串口通信: JSON 协议，PC 发送 Profile 到 ESP32，ESP32 解析后更新屏幕
  • BLE GATT: 128-bit UUID 服务，支持长写入（Prepare + Execute）
  • ESP32 UI: 动态显示 Profile 名称和按键映射，10Hz 刷新率

产品规划要点

  • 核心流程: 软件识别 → 工作流识别 → 自动适配 → 行为记录
  • 本月目标: 个人工作流数据库（SQLite 行为日志 + 工作流分析）
  • AI 预留: 接口定义完成，先用规则引擎，后接 LLM API
  • 4周计划: Week1 核心链路 → Week2 行为记录 → Week3 工作流分析 → Week4 打磨
